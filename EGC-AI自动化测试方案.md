## 1. 方案总览

### 核心闭环

```
触发 → 环境准备 → 逐用例循环校验 → 报告归档
```

基于 AI 测试用例生成 + 双 MCP 服务架构 + IEC104 模拟器，实现从**触发执行 → 环境准备 → 逐用例循环校验 → 报告输出 → 归档留痕**的全链路闭环回归测试。

### 整体架构

```
                    ┌─────────────────────────┐
                    │     测试触发器            │  触发方式：定时/手动/CI
                    │  (指定场景/回归套件)       │
                    └───────────┬─────────────┘
                                │ 指定场景
                                ▼
                    ┌─────────────────────────┐
                    │   EGC 策略配置器          │  待考虑
                    │   (按场景切换策略)         │  按场景加载对应策略模板
                    └───────────┬─────────────┘
                                │ 环境就绪
                                ▼
                    ┌─────────────────────────┐
                    │   AI 测试用例生成器        │  已有
                    │   (场景→结构化用例)        │
                    └───────────┬─────────────┘
                                │ 描述
                                ▼
                    ┌─────────────────────────┐
                    │   测试数据生成器           │  待实现
                    │   (用例→CSV时序数据)       │  列名=IOA地址，Agent内建映射
                    └───────────┬─────────────┘
                                │ CSV
                                ▼
                    ┌─────────────────────────┐
                    │   双 MCP 服务             │  待实现
                    │   Instance Manager       │  管理面：实例生命周期
                    │   Data Interface         │  数据面：读写/策略/CSV/QDS
                    └───────────┬─────────────┘
                                │ HTTP API
                                ▼
                    ┌─────────────────────────┐
                    │   IEC104 模拟器           │  已有
                    │   (模拟关口表/光伏/储能)    │
                    └───────────┬─────────────┘
                                │ IEC104 协议
                                ▼
                    ┌─────────────────────────┐
                    │   EGC (Edge Grid         │  已有（被测系统）
                    │   Controller)            │
                    └───────────┬─────────────┘
                                │ 读取 AO 控制指令
                                ▼
                    ┌─────────────────────────┐
                    │   归档 & 报告仓库         │  待实现
                    │   (全量留存/回归对比)      │
                    └─────────────────────────┘
```

### 核心思路

- **闭环自动化**: AI 生成测试用例 → 自动展开为时序数据 → 双 MCP 服务将数据写入 IEC104 模拟器 → EGC 感知数据变化并下发控制指令（写入 AO/DO 点）→ MCP 读取控制结果 → 与预期结果比对 → 生成测试报告 → 全量归档。
- **测试即存档**: 每次执行的所有产物全量归档，下一次回归时可选择任意历史场景重新执行，对比结果趋势。
- **触发方式**: 支持定时任务（如每日凌晨回归）、手动触发（指定场景）、CI/CD 事件触发（代码提交后自动执行）。
- **Agent 自治**: AI Agent 负责从触发到归档的全流程编排，包括数据生成、映射解析、执行校验、报告输出。

## 2. 为什么需要自动化

### 手动测试效率低
每次测试需要手动配置 EGC 策略、在模拟器上逐点设置测点值（AI/DI/PI 不同值字段）、等待 EGC 响应后再核对 AO 控制结果。一个场景耗时 10-15 分钟，且容易出错。

### 覆盖度不足
边界条件、异常品质（QDS 无效/替代）、策略组合、时序场景等因手工成本高被忽略。回归测试只能覆盖核心路径，风险漏测普遍存在。

### 无法有效回归
测试结果没有系统化留存，每次回归都是重新执行，无法对比历史趋势。问题复现时需要重复搭建环境，排查效率低下。

## 3. 自动化执行四步闭环

### 步骤 1：触发 & 指定场景

**触发方式**:
| 方式 | 说明 |
|------|------|
| **定时任务** | 每日凌晨/每周执行全量回归套件 |
| **手动触发** | 指定场景名称，立即执行该场景的所有用例 |
| **事件驱动** | EGC 版本发布后自动触发对应回归套件 |

**指定范围**:
- **初版**: 按功能模块选择场景（如"并网点限功率"、"储能目标跟踪"）
- **后续**: 整合为版本级回归套件，一键执行全部场景

触发时同时确定 **EGC 控制策略配置**（不同场景需要不同的策略参数）。

### 步骤 2：环境准备

| 子步骤 | 内容 |
|--------|------|
| **a. EGC 控制策略配置** | 根据测试场景，设置 EGC 的对应控制策略参数（如目标功率限值、SOC 上下限、控制模式等）。策略配置的快照将随本次测试一并归档 |
| **b. 测试数据准备** | 从归档中读取已有测试数据，或由 AI Agent 按测试用例即时生成 CSV 格式的时序数据。数据包含 IOA 地址列名，直接兼容模拟器 |
| **c. 环境就绪检查** | 确认 IEC104 模拟器已启动、EGC 在线、MCP 服务健康 |

### 步骤 3：逐用例循环校验

针对本场景中的每一条测试用例，按顺序执行：

```
① 写入输入数据 ──→ ② 等待 EGC 控制周期 ──→ ③ 读取控制结果 ──→ ④ 比对 & 记录
```

| 阶段 | 操作 | 说明 |
|------|------|------|
| **① 写入输入数据** | `MCP write_points` | 一次性写入该用例的所有输入测点（关口表、光伏、储能、SOC 等） |
| **② 等待控制周期** | 等待 EGC | EGC 完成数据采集 → 策略计算 → AO/DO 指令下发 |
| **③ 读取控制结果** | `MCP read_points` | 读取 EGC 写入 AO 点的实际控制指令（光伏目标、储能目标、并网点功率） |
| **④ 比对 & 记录** | 预期 vs 实际 | 支持容差，通过/失败记录到运行报告中。不通过则记录，继续执行下一条 |

└─ **循环结束条件**: 该场景所有用例执行完毕

### 步骤 4：分析 → 报告 → 归档

| 子步骤 | 内容 |
|--------|------|
| **a. 生成报告** | 汇总执行结果：通过率、每用例明细（实际值 vs 预期值、偏差量）、失败用例分布、执行耗时统计 |
| **b. 推送报告** | 报告自动推送（邮件/即时通讯/网页），相关人员即时知晓测试结果 |
| **c. 全量归档** | 测试数据 CSV、EGC 策略配置快照、MCP 执行日志、原始结果数据、最终报告（PDF/HTML），按执行批次留存 |
| **d. 回归对比** | 同一场景的历次报告可对比趋势，发现性能退化或行为变更 |

### EGC 控制策略配置说明

当前阶段将 EGC 策略配置作为**待考虑项**记录在方案中。不同测试场景确实需要不同的 EGC 策略参数（如并网点限功率场景 vs 储能目标跟踪场景的策略配置不同）。

初步思路：
- 将 EGC 策略配置模板化，每个场景关联一个策略配置文件
- 环境准备步骤根据场景名称加载对应的策略配置并下发到 EGC
- 策略配置的修改接口取决于 EGC 的实现方式（配置文件热加载、API 动态调整、或通过模拟器 IOA 写入特定策略测点）

**实施前提**: 需要明确 EGC 提供何种方式让外部修改控制策略配置。

## 5. IEC104 测点类型体系

| 类型 | 含义 | 值字段 | 方向 | 说明 |
|------|------|--------|------|------|
| **AI** | 遥测 | `value` (float64) | 读写 | 关口表功率、光伏功率、SOC 等模拟量 |
| **DI** | 遥信 | `bool_value` (bool) | 读写 | 断路器状态、设备启停信号 |
| **PI** | 遥脉 | `int_value` (int32) | 读写 | 电能量脉冲计数 |
| **AO** | 遥调 ← EGC 下发 | `value` (float64) | 只读(API) | 光伏/储能目标功率 — 自动化引擎读取此值做校验 |
| **DO** | 遥控 ← EGC 下发 | `bool_value` (bool) | 只读(API) | 设备启停控制指令 |

### 关键约束

- **写入约束**: AI/DI/PI 可通过 HTTP API 写入；AO/DO 由 EGC 通过 IEC104 协议写入，HTTP API **禁止写入**。
- **读取无限制**: 所有类型都可通过 HTTP API 读取，自动化引擎读取 AO 点校验 EGC 控制指令完全可行。
- **自动变化策略**: 当测点启用自动变化且策略非 `APIUpdate` 时，HTTP API 拒绝写入。
- **Agent 感知**: AI Agent 在生成测试数据时已内建测点类型的知识，MCP 层在收到写入请求时根据 IOA 自动选择正确的值字段。

## 6. 测试数据格式

### CSV 示例

| 时间戳 | 40001 | 40002 | 40003 | 40004 | 40005 | 30001 | 30002 |
|--------|-------|-------|-------|-------|-------|-------|-------|
| 类型 | 关口表_P(AI) | 关口表_Q(AI) | 光伏_P(AI) | 储能_P(AI) | 储能_SOC(AI) | 预期_光伏目标(AO) | 预期_储能目标(AO) |
| 方向 | **📥 输入** | 输入 | 输入 | 输入 | 输入 | **✅ 预期输出** | 预期输出 |
| T0 | 50 | 0 | 30 | -10 | 60 | 30 | -10 |
| T1 | 120 | 10 | 60 | -20 | 55 | 45 | 25 |
| T2 | 80 | -5 | 40 | 10 | 45 | 30 | 0→20 |
| T3 (QDS异常) | 200 | 15 | 80 | 30 | 80 | 0 | 100 |

### 关键设计：无额外映射层

**列名=IOA 地址**: CSV 列头直接使用模拟器的 IOA 地址（40001、40002...），生成的数据 CSV 可以直接导入模拟器，也可以被 MCP `write_points` 直接消费。

**Agent 内建映射**: AI Agent 在生成测试数据时已理解 IOA ↔ 逻辑名称 ↔ 测点类型的对应关系。测试数据本身就是知识的载体，不需要再维护一份额外的 YAML 映射文件。

**与模拟器无缝衔接**: 模拟器已有 CSV 导入/导出能力，测试数据生成器产出的 CSV 就是模拟器原生格式，**生成即消费**。

**QDS 支持**: 如需注入异常品质，可在 CSV 中增加 `40001_qds` 列，MCP 层自动识别并将其附着到对应值的 QDS 字段。

## 7. 双 MCP 服务设计

### MCP Instance Manager

模拟器实例生命周期管理。测试准备阶段确保模拟器就绪，执行结束后可清理实例。

| 工具名 | 描述 | 类型 |
|--------|------|------|
| `list_instances` | 列出所有已配置实例 | READ |
| `get_instance` | 获取单个实例详情 | READ |
| `create_instance` | 创建新的模拟器实例 | WRITE |
| `delete_instance` | 删除指定实例 | WRITE |
| `start_instance` | 启动实例，开始 IEC104 通信 | WRITE |
| `stop_instance` | 停止实例，断开连接 | WRITE |
| `restart_instance` | 重启实例 | WRITE |
| `get_server_status` | 模拟器整体健康度 | READ |

**通信方式**: HTTP → IEC104 Simulator REST API

### MCP Data Interface

所有测点数据操作。Agent 直接传入 IOA 地址和值，MCP 层自动处理类型转换和品质描述符附着。

| 工具名 | 描述 | 类型 |
|--------|------|------|
| `read_point` | 按 IOA 读取当前值，自动返回类型感知结果 | READ |
| `read_points` | 一次读取多个 IOA。用于校验 EGC 下发的 AO 控制指令 | READ BATCH |
| `write_point` | 向指定 IOA 写入值。自动选择值字段，可选 QDS | WRITE |
| `write_points` ⭐ | **一次写入多个 IOA。输入数据 CSV 的列可直接映射为此调用的参数** | WRITE BATCH |
| `list_points` | AI 优先、IOA 升序，含当前值 | READ |
| `config_auto_change` | 为指定 IOA 配置自动变化方式 | WRITE STRATEGY |
| `batch_auto_change` | 批量配置自动变化 | WRITE BATCH |
| `get_auto_change` | 查看自动变化配置 | READ |
| `delete_auto_change` | 删除自动变化配置 | WRITE |
| `export_auto_changes` | 导出全部自动变化配置(CSV) | READ BATCH |
| `import_auto_changes` | 导入自动变化配置(CSV) | WRITE BATCH |
| `upload_csv` | 上传 CSV 回放文件，结合 StrategyCSV 实现时序数据自动播放 | WRITE CSV |
| `export_points_csv` | 导出测点数据(CSV) | READ |

**通信方式**: HTTP → IEC104 Simulator REST API

### MCP 架构要点

- **职责清晰**: 管理面（Instance Manager）与数据面（Data Interface）完全分离
- **Agent 传 IOA 直写**: AI Agent 在测试数据中已嵌入 IOA 地址，调用 `write_points` 时直接传入 `{"40001": 50, "40003": 30}`，MCP 层通过 IOA 查类型并自动选择值字段。**无需额外的 YAML 映射文件**
- **Agent 内建映射知识**: AI Agent 生成测试数据时，已经理解了每个 IOA 对应的设备、类型和业务含义

## 8. 混合执行模式

### 模式 A：MCP 精确控制（推荐）

`write_points` 写入 → 等待 EGC → `read_points` 校验

| 优点 | 缺点 |
|------|------|
| 每个时间步可控写入和读取时机 | 依赖 `POST /points/batch` 端点 |
| 可逐步骤校验，精确定位失败点 | |
| 支持 QDS 附加和异常注入 | |

**适用于**: 需要精确校验的场景

### 模式 B：CSV 回放简化

模拟器内置 `StrategyCSV`，上传 CSV 后自动按时间轴播放

| 优点 | 缺点 |
|------|------|
| 无需批量端点，模拟器自动处理时序 | 无法在每步之间同步校验 |
| 减少 MCP 调用次数 | 需要事后再批量读取 AO 点比对 |

**适用于**: 长时间稳定性测试 / 纯数据驱动场景

### 混合使用建议

- **模式 A** 覆盖绝大多数测试场景（精确校验每一步的控制结果）
- **模式 B** 用于长时间稳定性测试或模拟器底层性能验证
- **两个模式可共存**: 同一实例中部分测点 CSV 自动播放，其他测点 MCP 手动控制

## 9. 归档与回归体系

### 归档结构

```
archive/
  2026-05-13_1430_scenario-grid-limit/    # 执行批次：日期+时间+场景名
    ├── config/
    │   ├── egc_strategy.yaml             # EGC 策略配置快照
    │   └── simulator_instance.json       # 模拟器实例配置
    ├── data/
    │   ├── test_data.csv                 # 原始测试数据
    │   └── raw_results.csv               # 每步实际读取值
    ├── logs/
    │   ├── mcp_execution.log             # MCP 调用日志
    │   └── engine.log                    # 执行引擎日志
    └── report/
        ├── report.html                   # 可视化报告
        └── report.json                   # 结构化数据
```

### 全量留存

- 测试数据 CSV（原始输入）
- EGC 策略配置快照
- MCP 执行日志
- 原始结果数据（每步的实际读取值）
- 最终报告（HTML + JSON 双格式）
- 产物按批次整理，永不覆盖

### 回归应用

- 选择任一历史场景 → **一键重新执行**
- 相同场景的历次报告可**并排对比**
- 对比项：通过率、平均偏差、执行时长
- 初版：按功能模块选择场景
- 成熟期：整版本回归套件一键全跑
- 差异告警：通过率下降 → 自动通知

|------|------|
| 修改方式 | 需要明确 EGC 提供何种方式修改控制策略 |
| 可能方式 | 配置文件热加载 / API 接口 / 通过 IOA 写入 |
| 模板关联 | 策略模板与场景的关联方式 |
| **此依赖确定后** | 策略配置器模块才能实现 |

### Phase 0.5 前置依赖

Phase 1 的 `write_points` 工具依赖于模拟器新增的 `POST /points/batch` 端点。若该端点未实现，则退化为循环调用 `write_point`（不保证原子性），测试结果的数据完整性风险由测试人员评估。
